# 业务域管理开发思路

## 当前方案：以 Settings 为准

### 核心设计

业务域（Domain）的权威来源是 `settings/` 目录下的子文件夹。每个业务域对应三个目录：

```
input/{domain}/         # Markdown 源文件（用户维护）
settings/{domain}/      # 配置和缓存（API 管理）
output/{domain}/        # 构建输出（自动生成）
```

### Domain 识别逻辑

```go
// ListDomains 遍历 settings/ 目录
func (dm *DomainManager) ListDomains() ([]string, error) {
    entries, err := os.ReadDir(dm.domainsDir)  // cfg.SettingsDir
    for _, entry := range entries {
        if entry.IsDir() {
            domains = append(domains, entry.Name())
        }
    }
}
```

**设计理由**：
- 简单直接，无需额外数据库
- 配置（settings）是业务域的核心，input 可以没有（新建 domain 时），output 可以没有（未构建时）
- 与 `--settings` 参数语义一致

### 配置文件关系

```
settings/{domain}/
├── domain_config.json    # 业务域主配置（API 读写）
│   ├── domain_name       # 域名
│   ├── display_name      # 显示名
│   ├── description       # 描述
│   ├── root_folders      # 一级目录配置（U3）
│   ├── aggregation_fields # 聚合字段（图谱/目录）
│   └── build_overrides   # 构建设置覆盖
│
├── config.json           # Quartz 配置（自动生成）
│   ├── baseUrl           # 覆盖 quartz.config.ts
│   └── graph             # 图谱聚合配置
│
└── layout.json           # 布局配置（手动维护或未来扩展）
    ├── explorer          # 目录配置
    ├── backlinks         # 反向链接配置
    └── folderPage        # 文件夹页面配置
```

### API 设计

```
GET    /api/domain                    # 列出所有业务域
POST   /api/domain/create             # 创建业务域
GET    /api/domain/{domain}/config    # 获取配置
PUT    /api/domain/{domain}/config    # 更新配置
DELETE /api/domain/{domain}/config    # 删除配置
POST   /api/domain/{domain}/build     # 触发构建
```

### 构建流程

```
1. POST /api/domain/{domain}/build
   ↓
2. Load domain_config.json
   ↓
3. Generate config.json (ApplyToQuartzConfig)
   ↓
4. Run: quartz build --settings={settingsDir}/{domain} -d {inputDir}/{domain} -o {outputDir}/{domain}
   ↓
5. Output generated to output/{domain}/
```

### 与 Client 的协作

**Client (Quartz/TypeScript)**：
- 通过 `--settings={path}` 指定配置目录
- 读取 `{path}/config.json` 覆盖 quartz.config.ts
- 读取 `{path}/layout.json` 覆盖布局配置
- 在 `{path}/` 创建 `.quartz-cache.db` 图谱缓存

**Server (Go)**：
- 管理 `domain_config.json`
- 自动生成 `config.json`
- 未来可扩展生成 `layout.json`

### 注意事项

1. **手动创建 input 目录**：
   - 如果直接在 `input/` 下创建新目录，API 不会自动识别
   - 需要先调用 `POST /api/domain/create` 创建 domain
   - 或者后续做自动发现（扫描 input/ 并自动注册缺失的）

2. **config.json 覆盖范围**：
   - 目前 Server 只生成 `baseUrl` 和 `graph` 字段
   - 如需更多字段，需在 `generateQuartzConfig()` 中扩展

3. **layout.json 管理**：
   - 目前 Server 不生成 layout.json
   - 如需后端管理布局，需扩展 `ApplyToQuartzConfig()` 生成 layout.json

### 未来扩展方向

| 优先级 | 功能 | 实现方式 |
|--------|------|----------|
| P1 | 更多 config 字段 | 扩展 `generateQuartzConfig()` |
| P2 | 生成 layout.json | 在 `ApplyToQuartzConfig()` 中增加 layout 生成 |
| P3 | 构建历史记录 | 在 domain_config.json 中增加字段，或单独文件 |
| P4 | Domain 自动发现 | 启动时扫描 input/，发现未注册的自动创建 settings |

### 为什么不选其他方案

**方案 A（以 Input 为准）**：
- 问题：新建 domain 时 input 可能为空，无法判断是 domain 还是临时文件夹
- 结论：配置才是 domain 的本质

**方案 D（SQLite）**：
- 问题：引入额外依赖，需要处理 db 与目录的同步
- 结论：当前需求简单，文件系统足够，避免过度设计
