# 业务域管理开发思路

## 当前方案：以 Settings 为准

### 核心设计

业务域（Domain）的权威来源是 `settings/` 目录下的子文件夹。每个业务域对应三个目录：

```
input/{domain}/         # Markdown 源文件（用户维护）
settings/{domain}/      # 配置和缓存（API 管理）
output/{domain}/        # 构建输出（自动生成）
```

### Domain 识别逻辑

```go
// ListDomains 遍历 settings/ 目录
func (dm *DomainManager) ListDomains() ([]string, error) {
    entries, err := os.ReadDir(dm.domainsDir)  // cfg.SettingsDir
    for _, entry := range entries {
        if entry.IsDir() {
            domains = append(domains, entry.Name())
        }
    }
}
```

**设计理由**：
- 简单直接，无需额外数据库
- 配置（settings）是业务域的核心，input 可以没有（新建 domain 时），output 可以没有（未构建时）
- 与 `--settings` 参数语义一致

### 配置文件关系

```
settings/{domain}/
├── quartz.config.json      # Quartz 配置（Server 管理）
│   ├── pageTitle           # 显示名（用户可设置）
│   ├── baseUrl             # 由服务器自动生成（不可用户自定义）
│   └── graph               # 图谱聚合配置
│
└── quartz.layout.json      # 布局配置（Server 管理）
    ├── backlinks           # 反向链接配置
    └── aggregation          # 聚合分组配置
```

**配置映射**：
| 概念 | 来源 | 说明 |
|------|------|------|
| Domain ID | settings 子目录名 | 自动获取 |
| Display Name | `quartz.config.json` 的 `pageTitle` | 用户可设置 |
| Base URL | 服务器自动生成 | 由 `server/config.json` 的 `base_url` + domain 拼接，**不允许用户自定义** |
| Graph 配置 | `quartz.config.json` 的 `graph` | 用户可设置 |
| Layout 配置 | `quartz.layout.json` | 用户可设置 |

### API 设计

```
GET    /api/domains                   # 列出所有业务域
POST   /api/domain/create             # 创建业务域
GET    /api/domain/{domain}            # 获取业务域信息
PUT    /api/domain/{domain}            # 更新业务域配置（config + layout）
DELETE /api/domain/{domain}            # 删除业务域
POST   /api/domain/{domain}/build      # 触发构建
GET    /api/domain/{domain}/status     # 获取构建状态
GET    /api/domain/{domain}/logs      # 获取构建日志
GET    /api/domain/tasks              # 获取所有运行中的任务
```

### 构建流程

```
1. POST /api/domain/{domain}/build
   ↓
2. Load quartz.config.json + quartz.layout.json
   ↓
3. Generate baseUrl (from server base_url + domain)
   ↓
4. Run: quartz build --settings={settingsDir}/{domain} -d {inputDir}/{domain} -o {outputDir}/{domain}
   ↓
5. Output generated to output/{domain}/
```

### 与 Client 的协作

**Client (Quartz/TypeScript)**：
- 通过 `--settings={path}` 指定配置目录
- 读取 `{path}/quartz.config.json` 覆盖 quartz.config.ts
- 读取 `{path}/quartz.layout.json` 覆盖布局配置
- 在 `{path}/` 创建 `.quartz_cache.db` 图谱缓存

**Server (Go)**：
- 管理 `quartz.config.json` 和 `quartz.layout.json`
- 自动生成 `baseUrl`
- 不需要 `domain_config.json` 中间层

### 注意事项

1. **手动创建 input 目录**：
   - 如果直接在 `input/` 下创建新目录，API 不会自动识别
   - 需要先调用 `POST /api/domain/{domain}` 创建 domain
   - 或者后续做自动发现（扫描 input/ 并自动注册缺失的）

2. **baseUrl 由服务器控制**：
   - 用户在 API 请求中传入的 `baseUrl` 会被忽略
   - baseUrl = `{server config.json base_url}/{domain}`

3. **layout.json 管理**：
   - 通过 `PUT /api/domain/{domain}` 的 `layout` 字段更新
   - 结构与 quartz.layout.ts 对齐

### 未来扩展方向

| 优先级 | 功能 | 实现方式 |
|--------|------|----------|
| P1 | 更多 graph 配置字段 | 扩展 QuartzConfig 结构 |
| P2 | 构建历史记录 | 独立日志文件，已支持 |
| P3 | Domain 自动发现 | 启动时扫描 input/，发现未注册的自动创建 settings |

### 为什么不选其他方案

**方案 A（以 Input 为准）**：
- 问题：新建 domain 时 input 可能为空，无法判断是 domain 还是临时文件夹
- 结论：配置才是 domain 的本质

**方案 D（SQLite）**：
- 问题：引入额外依赖，需要处理 db 与目录的同步
- 结论：当前需求简单，文件系统足够，避免过度设计
