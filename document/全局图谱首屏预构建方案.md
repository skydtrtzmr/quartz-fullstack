# 全局图谱首屏预构建方案

## 背景

全局图谱首次渲染时，需要加载完整的 `contentIndex.json`（可能包含数万个节点的元数据），然后在前端执行以下计算：

1. 构建链接图（解析所有节点的出链）
2. 计算连接数，筛选核心节点
3. 执行聚合规则（按文件夹/字段/日期分组）
4. 大区分组（如果配置了 `regionRules`）
5. 初始化 D3 力导向图模拟

这些计算在节点数量大时会导致首屏加载缓慢，用户体验不佳。

## 目标

- **首屏时间 < 500ms**：用户打开全局图谱后立即看到图
- **不影响后续交互**：用户展开节点、拖拽、缩放等功能不受限
- **渐进加载**：先渲染核心内容，再异步补充完整数据

## 方案概述

采用**渐进加载**策略：

```
阶段1（0-100ms）     →   阶段2（100ms-2s）    →   阶段3（完成）
快照渲染首屏          →   异步加载完整数据      →   增量补充，平滑过渡
   ↓                       ↓                       ↓
用户立即看到核心节点图    后台计算完整图谱          新节点渐入，布局稳定
```

## 详细设计

### 1. 构建时：生成首屏快照

**快照生成逻辑**（与前端保持一致）：

1. 读取 `contentIndex.json`
2. 构建链接图，计算每个节点的连接数
3. 应用 `coreNodeFilter` 规则筛选核心节点
4. 应用 `coreNodeLimit` 硬上限
5. 应用 `aggregation` 规则生成聚合节点
6. 应用 `regionRules` 生成大区节点（如果配置）
7. 提取首屏可见的节点和边

### 2. 前端：渐进加载

**修改文件**：`client/quartz/components/scripts/graph2.inline.ts`

#### 阶段1：快照渲染首屏

```typescript
async function initGraph() {
  const snapshot = await fetchGlobalSnapshot(basePath)
  
  let graphData: { nodes: NodeData[], links: LinkData[] }
  
  if (snapshot) {
    // 用快照初始化首屏（极快，~10-50KB）
    graphData = buildGraphFromSnapshot(snapshot)
    renderGraph(graphData)
    
    // 后台异步加载完整数据
    loadFullDataAsync(snapshot, graphData)
  } else {
    // 无快照，走现有逻辑
    graphData = await buildGraphFromContentIndex()
    renderGraph(graphData)
  }
}
```

#### 阶段2：异步加载完整数据

```typescript
async function loadFullDataAsync(snapshot, currentGraphData) {
  // 异步加载 contentIndex（可能较大）
  const contentData = await fetchContentIndex()
  
  // 计算完整图谱（与现有逻辑相同）
  const fullGraph = computeFullGraph(contentData)
  
  // 增量合并
  mergeGraphIncrementally(currentGraphData, fullGraph)
}
```

#### 阶段3：增量合并（平滑过渡）

```typescript
function mergeGraphIncrementally(current, full) {
  const existingIds = new Set(current.nodes.map(n => n.id))
  
  // 找出快照中未包含的节点
  const newNodes = full.nodes.filter(n => !existingIds.has(n.id))
  
  // 找出新增的边（两端都在合并后图中的边）
  const currentIds = new Set(current.nodes.map(n => n.id))
  const newLinks = full.links.filter(l => 
    currentIds.has(l.source.id) && currentIds.has(l.target.id)
  )
  
  // 平滑添加新节点
  newNodes.forEach(node => {
    // 从中心位置渐入，避免剧烈布局变化
    node.x = centerX + (Math.random() - 0.5) * 50
    node.y = centerY + (Math.random() - 0.5) * 50
    node.opacity = 0
    current.nodes.push(node)
    
    // 渐入动画
    animate(node, { opacity: 1 }, 500)
  })
  
  newLinks.forEach(link => current.links.push(link))
  
  // 重启力导向模拟，但使用低 alpha 值
  // 现有节点位置基本保持不变，新节点逐渐找到位置
  simulation.nodes(current.nodes)
  simulation.force("link").links(current.links)
  simulation.alpha(0.3).restart()
}
```

### 3. 交互兼容性

**快照节点与完整数据的映射**：

- 快照中的节点 ID 与完整数据一致，展开时直接从完整数据中查找子节点
- 聚合节点、大区节点的展开逻辑不变
- 用户点击"展开全部"时，如果完整数据已加载，立即显示；否则等待加载完成

**状态管理**：

```typescript
enum GraphDataState {
  SnapshotOnly,      // 仅快照数据
  LoadingFull,       // 正在加载完整数据
  FullDataReady,     // 完整数据已加载但未合并
  FullyMerged        // 完整数据已合并
}
```

## 性能对比

| 指标 | 纯 contentIndex | 纯快照 | 渐进加载（本方案） |
|------|----------------|--------|------------------|
| 首屏时间 | 2-10s（取决于数据量） | <100ms | <100ms |
| 首屏节点数 | 全部 | 50-100 核心节点 | 50-100 核心节点 |
| 首屏可交互 | 是 | 有限（展开需加载） | 是（最终完整） |
| 内存占用 | 高（全量数据） | 低 | 高（最终全量） |
| 网络请求 | 1 次（大文件） | 1 次（小文件） | 2 次（小+大） |
| 用户体验 | 白屏等待 | 可能闪烁/二次加载 | 平滑过渡 |

## 注意事项

1. **增量构建**：快照 Emitter 需要支持 `partialEmit`，在增量构建时只更新变化的节点
2. **版本兼容性**：快照数据结构需要版本号，前端加载时检查版本是否匹配
3. **配置变更**：当 `coreNodeFilter`、`regionRules`、`aggregation` 等配置变更时，需要重新生成快照（可通过配置哈希判断）
4. **降级方案**：如果快照加载失败或不存在，自动回退到现有逻辑
