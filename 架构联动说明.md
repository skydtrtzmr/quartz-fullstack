# Quartz 全栈架构与文件夹联动说明

本项目的核心设计理念是**“前端引擎层”与“业务数据层”的解耦**。
通过统一的后端调度，配合前段支持动态参数，实现一套 Quartz 代码支持无限个独立业务域的站点构建。

## 目录与参数映射关系

整个架构依赖以下几个核心文件夹路径，它们在服务端的 `config.json` 中配置，并通过后端 API 转化为前端 Quartz CLI 的运行参数：

| 逻辑概念 | 服务端配置 (config.json) | 默认路径示例 | 作用概述 | 传递给前端的命令参数 |
|:---|:---|:---|:---|:---|
| **输入源** | `input_dir` | `./input` | 存放所有 Markdown 原始文件的根目录。按业务域分子目录存放。 | `-d <input_dir>/<domain>` |
| **输出池** | `output_dir` | `./output` | 存放最终编译生成的 HTML 静态网站。按业务域分子目录存放。 | `-o <output_dir>/<domain>/public` |
| **动态配置** | `settings_dir` | `./settings` | 存放各业务域特定的 `quartz.config.json` 和 `quartz.layout.json` 覆写文件。 | `--settings <settings_dir>/<domain>` |
| **图谱缓存** | （自动推导） | `<output_dir>/<domain>/.cache` | 存放 SQLite 解析结果增量缓存，确保各域的增量构建互不干扰。 | `--cacheDir <output_dir>/<domain>/.cache` |

## 运行时数据流转模型

### 1. 结构概览

```
workspace_root/
  ├─ client/                # (源码) Quartz 前端代码，纯粹的解析与渲染引擎
  ├─ server/                # (源码) Go 后端服务，负责接口路由与构建任务调度
  │   └─ config.json        # 预设各种运行文件夹的绝对路径基础值
  │
  ├─ input/                 # (数据) Input_Dir (业务 MD 文件)
  │   ├─ xm/                # domain: xm 的文档中心
  │   └─ docs/              # domain: docs 的文档中心
  │
  ├─ settings/              # (数据) Settings_Dir (布局与配置)
  │   ├─ xm/                # 针对 xm 域的纯数据覆写
  │   │   ├─ quartz.config.json   # 主题、标题等
  │   │   └─ quartz.layout.json    # 排序策略、分布选项等
  │   └─ docs/
  │
  └─ output/                # (结果) Output_Dir (构建产物与缓存)
      ├─ xm/
      │   ├─ public/        # 静态 HTML (最终 serve 的目录)
      │   └─ .cache/        # 该域的 SQLite 分析缓存 (.quartz-cache.db)
      └─ docs/
```

### 2. 构建调度流程 (Build Pipeline)

当后端 API `/api/build?domain=xm` 被触发时：
1. **获取基础路径**: 后端从自己的 `config.json` 读出配置的 `input_dir`、`output_dir`、`settings_dir`。
2. **拼接业务域路径**: 检测如果是指定了 `domain=xm`，则分别拼接出对应的子侧重路径。
3. **注入 CLI 执行**: 最终执行在 `client` 目录下执行如下命令（伪代码）：
   ```bash
   node ./quartz/bootstrap-cli.mjs build \
     -d {input_dir}/xm \
     -o {output_dir}/xm/public \
     --settings {settings_dir}/xm \
     --cacheDir {output_dir}/xm/.cache \
     --sqlite
   ```
4. **引擎处理 (前端运行时)**: 
   - `build.ts` 首先定位到 `--cacheDir` 恢复增量 SQLite，如果不存在则会重新建表。
   - `quartz.config.ts` 读取 `--settings` 动态合并标题/外观。
   - `quartz.layout.ts` 读取 `--settings` 并挂载对应排序算法的指针函数。
   - 引擎根据 `-d` 读取 MD 源文件，编译结束后输出到 `-o` 中。

### 3. 静态资源路由 (Serve Pipeline)

当用户通过浏览器访问 `http://ip:port/xm/index.html` 时：
1. Go HTTP 服务层拦截路径前缀 `/xm/`。
2. 提取出 `domain=xm`，将剩余 URL `index.html` 映射回本地文件系统。
3. 返回位于 `{output_dir}/xm/public/index.html` 的文件。

### 总结

这套方案真正做到了：“**引挚代码零污染、配置策略全剥离、部署路径可任意移动**”。您不仅可以在开发阶段统一看全貌，后期还可以随意将 `input` / `output` 挂载到独立硬盘或 S3 OSS 映射盘内，只需要在后端的 `config.json` 修改根路径即可，非常灵活！
